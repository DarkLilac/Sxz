Example of using a polyfill to emulate transparent JPEG with a Sxz stencil similar to CSS shapes from Image. A benefit of using Sxz over PNG to do the stencil is size reduction. For a 512x512 pixel image, PNG will store one byte per pixel in indexed mode, which after DEFLATE brings the PNG to about 18K. Since Sxz can store the stencil as a bit-plane instead of a byte-plane, the uncompressed Sxz is about 33K. Which when compressed with Winzip comes to around 2.1K, about 1/8th the size of transparent-indexed-PNG.

It is tempting to try and reduce the JPEG image by removing the unsued data from it or extend the Sxz format with a chunk that does JPEG like encoding. Such as by removing MCUs that are entirely outside the rendered area and create a non-rectangular JPEG image. One problem is that JPEG decoders are written for performance and the format is not conducive to allow irregular MCUs nor is it easily extendable. Another problem is seen in the response to WebP which while it generates better compression than JPEG, fails to LOOK like JPEG causing viewer dissonance. So any JPEG replacement will need to display the same compression artifcats that we have grown accustomed to. A better method in the short term may be break the image up into multiple rectangular pieces and convert those to JPEG with varying degrees of compression, then stitch the separate JPEG images together with a lossless editor such as JpegTrans. The out of view areas of the image can be highly compressed and the viewable areas with better quality.
